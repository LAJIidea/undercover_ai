# 谁是卧底 - AI vs 真人 Web游戏

## 目标描述

开发一个基于Web的"谁是卧底"多人对战游戏，支持4个AI玩家与4个真人玩家对决。游戏面向娱乐视频录制场景设计，电脑大屏展示游戏主界面，真人通过手机浏览器接入。AI玩家通过OpenRouter API接入多种大模型，AI主持人负责游戏流程控制和裁判。语音交互通过阿里CosyVoice（TTS）和FunASR（STT）实现。

## 验收标准

遵循TDD理念，每条标准包含正向和反向测试用例以确保可验证性。

- AC-1: 游戏支持完整的3轮对战流程
  - 正向测试（预期通过）：
    - 创建游戏后，系统正确初始化3轮对战，每轮自动切换观察队/游戏队角色
    - 第1轮AI队为游戏队、真人队为观察队，第2轮反转，第3轮再反转
    - 3轮结束后正确计算总分并展示胜负结果
  - 反向测试（预期失败/拒绝）：
    - 尝试在3轮结束后继续开始新的轮次 → 被拒绝
    - 尝试跳过某一轮 → 被拒绝

- AC-2: 每轮游戏遵循完整的阶段流程（词语分配→讨论→轮流提问→投票）
  - 正向测试：
    - 系统随机从词语库选取词语，观察队4人全部收到词语，游戏队仅随机1人（全知者）收到
    - 每次正式提问前有固定30-60秒讨论时段，讨论内容对观察队可见
    - 游戏队按固定顺序轮流向主持人提问，主持人只回答"是"或"否"
    - 提问结束后观察队进行投票，由队长做最终决策指定全知者
  - 反向测试：
    - 游戏队非全知者成员尝试在游戏开始时查看词语 → 无法看到
    - 非当前轮次的发言者尝试提交正式提问 → 被拒绝
    - 观察队在提问阶段尚未结束时尝试提交投票 → 被拒绝

- AC-3: 每轮提问阶段严格限制在7分钟内
  - 正向测试：
    - 计时器从第一个提问开始计时，7分钟到达后强制结束提问阶段
    - 计时器在大屏UI上实时显示剩余时间
    - 7分钟内游戏队可以不限次数地提问（在轮流发言规则下）
  - 反向测试：
    - 7分钟超时后游戏队尝试继续提问 → 被拒绝，自动进入投票阶段

- AC-4: 计分规则正确实现
  - 正向测试：
    - 游戏队在7分钟内猜出词语 → 游戏队 +1分
    - 观察队队长正确指认全知者 → 观察队 +1分
    - 游戏队猜出词语且观察队猜出全知者 → 双方各 +1分
    - 双方均未猜出全知者时，提问数量更少的队伍 +1分
  - 反向测试：
    - 游戏队超时未猜出词语 → 游戏队不得分
    - 观察队队长指认错误的人为全知者 → 观察队不得分

- AC-5: 通过OpenRouter API支持6个模型厂商
  - 正向测试：
    - 配置界面可为4个AI玩家分别选择ChatGPT、Claude、Gemini、DeepSeek、Qwen、Kimi中的任意模型
    - 使用不同模型配置的AI玩家能正常参与游戏对话
    - AI主持人也可通过OpenRouter API配置使用的模型
  - 反向测试：
    - 配置了不存在的模型名称 → 显示错误提示
    - OpenRouter API key未配置或无效 → 游戏无法启动，提示配置错误

- AC-6: AI Agent能根据角色（普通玩家/全知者/主持人/观察者）正确行为
  - 正向测试：
    - AI作为全知者时，采用平衡型策略：混合有价值的引导性提问和迷惑性普通提问
    - AI作为普通游戏队成员时，基于已有问答信息进行合理推理和提问
    - AI作为观察者时，分析讨论和提问模式来判断全知者身份
    - AI主持人对是非问题给出准确、一致的回答
    - 4个AI各有预设性格标签（如谨慎型、分析型等），同时根据游戏进程动态调整策略
  - 反向测试：
    - AI全知者不会在提问/讨论中直接透露词语内容
    - AI主持人不会给出"是"或"否"以外的回答
    - AI普通玩家不会表现出知道词语的行为

- AC-7: 电脑大屏UI采用上下对称布局，适合视频录制
  - 正向测试：
    - 上方展示AI队4人虚拟形象+状态，下方展示真人队4人头像+状态
    - 中间区域展示主持人、对话流、计时器、比分
    - 当前发言者有高亮标识，讨论内容和正式提问有视觉区分
    - 界面简洁美观，适合直接录屏作为视频素材
  - 反向测试：
    - 界面不会出现影响录制的调试信息或技术性UI元素

- AC-8: 真人通过手机浏览器接入游戏
  - 正向测试：
    - 手机端打开游戏URL后可加入房间，显示适配移动端的操作界面
    - 手机端点击"发言"按钮后通过FunASR进行语音识别，文字发送到游戏
    - AI的发言通过CosyVoice转为语音，在电脑大屏端播放
    - 手机端可看到自己的角色信息（全知者可看到词语）
  - 反向测试：
    - 未加入房间的手机访问 → 只能看到等待/加入界面
    - 非游戏队成员的手机端尝试发起提问 → 操作不可用

- AC-9: 词语库支持混合模式（预设+AI生成）
  - 正向测试：
    - 系统内置预设词语库，按类别（人物、食物、用品）分类存储
    - 可配置使用AI动态生成词语，生成的词语符合类别要求
    - 可配置预设与AI生成的比例
  - 反向测试：
    - 词语库为空且AI生成被禁用时 → 无法开始游戏，提示需要配置词语来源

## 路径边界

路径边界定义了实现质量和选择的可接受范围。

### 上界（最大可接受范围）

完整的纯Web响应式应用，包含：电脑大屏端完整游戏界面（上下对称布局、动画效果、音效反馈）、手机端完整交互界面（语音输入、角色信息展示、讨论区）、完善的AI Agent系统（主持人Agent、玩家Agent含全知者/普通/观察者三种模式、预设性格+动态策略）、CosyVoice TTS和FunASR STT完整集成、预设词语库+AI生成混合模式、OpenRouter API集成支持全部6个模型厂商的可配置界面、WebSocket实时通信、完整的3轮游戏流程和计分系统。

### 下界（最小可接受范围）

纯Web应用，电脑大屏端展示基础的上下对称布局游戏界面（文字为主、基础样式）、手机端可通过浏览器接入并进行语音输入、AI Agent能完成基础的角色行为（主持人回答是/否、全知者基础引导、普通玩家基础提问、观察者基础投票）、STT/TTS通过预留接口集成（可降级为纯文字模式）、预设词语库、OpenRouter API集成支持6个模型厂商、基础WebSocket通信、完整的3轮游戏流程和计分系统。

### 允许的选择

- 可以使用：
  - 前端框架：React、Vue、Svelte等主流前端框架
  - 后端框架：Node.js（Express/Fastify/Koa）、Python（FastAPI/Flask）等
  - 实时通信：WebSocket（推荐）、SSE
  - 样式方案：Tailwind CSS、CSS Modules、Styled Components等
  - 状态管理：任何与所选前端框架兼容的方案
  - 数据库：轻量级方案（SQLite、JSON文件）用于词语库存储
- 不可以使用：
  - Electron或其他桌面封装方案（必须是纯Web）
  - 非OpenRouter的LLM API调用方式
  - 非CosyVoice的TTS方案、非FunASR的STT方案（除降级文字模式外）

### 概念性方案

```
整体架构：
  浏览器客户端（响应式）
    ├── 电脑大屏视图：游戏主界面（上下对称布局）
    │   ├── 上方：AI队4人虚拟形象区
    │   ├── 中间：主持人+对话流+计时器+比分
    │   └── 下方：真人队4人头像区
    └── 手机视图：玩家操作界面
        ├── 角色信息（含词语展示，仅全知者可见）
        ├── 发言按钮（触发STT录音）
        └── 讨论区

  后端服务
    ├── WebSocket服务器：实时游戏状态同步
    ├── 游戏引擎：状态机管理游戏流程
    │   ├── 状态：等待加入 → 游戏配置 → 轮次开始 → 词语分配
    │   │   → 讨论阶段 → 提问阶段 → 猜词阶段 → 投票阶段 → 轮次结算 → 游戏结束
    │   └── 7分钟硬性计时器
    ├── AI Agent管理器
    │   ├── 主持人Agent：接收提问 → 判断是/否 → 返回回答
    │   ├── 玩家Agent（全知者模式）：平衡型策略prompt
    │   ├── 玩家Agent（普通模式）：推理+提问prompt
    │   └── 玩家Agent（观察者模式）：分析+判断prompt
    ├── OpenRouter API适配层：统一接口，支持6个模型厂商
    ├── TTS服务（CosyVoice）：AI文字 → 语音流
    └── STT服务（FunASR）：真人语音 → 文字

AI全知者策略（平衡型）设计思路：
  - 在prompt中设定：你知道答案是X，但你需要隐藏这一事实
  - 策略混合比例：约60%正常提问 + 40%引导性提问
  - 引导方式：不直接问"是不是X"，而是缩小范围（如知道是"苹果"，问"是不是水果"）
  - 偶尔故意问偏方向的问题来制造迷惑
  - 讨论时表现出"思考"和"不确定"的语气
```

## 依赖和顺序

### 里程碑

1. 基础架构搭建：项目初始化和核心框架
   - 阶段A：初始化Web项目结构（前端+后端），配置构建工具
   - 阶段B：实现WebSocket服务器和基础通信协议
   - 阶段C：设计并实现游戏状态机（核心游戏流程控制）

2. AI系统集成：Agent设计和LLM接入
   - 阶段A：实现OpenRouter API适配层，支持6个模型厂商配置
   - 阶段B：设计并实现AI主持人Agent（是非问题回答）
   - 阶段C：设计并实现AI玩家Agent（全知者/普通/观察者三种模式+预设性格+动态策略）
   - 阶段D：实现AI模型配置界面（为4个AI玩家分别配置模型）

3. 游戏核心逻辑：完整的对战流程
   - 阶段A：实现词语库系统（预设词语库+AI生成混合模式）
   - 阶段B：实现轮次管理（3轮切换、角色分配、全知者随机选择）
   - 阶段C：实现提问流程（轮流发言、讨论时段、7分钟硬性计时）
   - 阶段D：实现投票和计分系统（队长决策、4条计分规则）

4. UI界面开发：大屏和手机端
   - 阶段A：电脑大屏上下对称布局（AI队/真人队/主持人/对话流/计时器/比分）
   - 阶段B：手机端响应式界面（加入房间、角色信息、发言按钮、讨论区）
   - 阶段C：视觉效果优化（发言高亮、讨论/提问区分、美观简洁适合录制）

5. 语音系统集成：STT和TTS
   - 阶段A：集成FunASR实现手机端语音转文字
   - 阶段B：集成CosyVoice实现AI文字转语音在大屏端播放
   - 阶段C：实现纯文字降级模式（当语音服务不可用时）

里程碑1是所有后续工作的基础。里程碑2和里程碑3可以并行推进（AI系统和游戏逻辑相对独立）。里程碑4依赖里程碑1的WebSocket通信和里程碑3的游戏流程。里程碑5可以在里程碑4之后或与里程碑4并行（先以文字模式开发，后叠加语音）。

## 实现备注

### 代码风格要求
- 实现代码和注释中不得包含计划文档特有术语，如"AC-"、"里程碑"、"阶段"、"步骤"等工作流标记
- 这些术语仅用于计划文档，不应出现在最终代码中
- 代码中使用描述性的、符合领域语义的命名

### 环境配置
- 项目根目录需要 `.env` 文件，包含以下配置项：
  - `OPENROUTER_API_KEY` - OpenRouter API密钥
  - `COSYVOICE_API_KEY` - 阿里CosyVoice TTS API密钥
  - `FUNASR_API_KEY` - FunASR STT API密钥
  - 其他必要的服务配置

### AI Agent 设计要点
- 全知者Agent采用"平衡型"策略：混合引导性提问和迷惑性提问，避免暴露身份
- 每个AI有预设性格标签，同时根据游戏进程动态调整策略
- 主持人Agent需确保回答的准确性和一致性，仅回答"是"或"否"
- AI在讨论阶段的发言应自然，避免机械感
